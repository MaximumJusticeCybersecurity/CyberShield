# TrustMap Memory Anchor Requirements

Date: 2026-05-28
Baseline: V51.1

## Purpose

TrustMap is the flagship conversion surface for CyberShield.  It must become the memorable executive visualization that makes leaders think:

> We probably do not currently have enough operational visibility into cyber and AI decision-making.

This is the rent-free memory object CyberShield should create.

## Strategic role

TrustMap is not a decorative diagram.  It is the symbolic operating language of CyberShield.

It should help executives see:

- fragmented visibility
- governance blind spots
- AI drift
- ownership ambiguity
- vendor concentration
- data movement risk
- authenticity risk
- evidence gaps
- decision accountability
- operational consequence

## Visual doctrine

TrustMap should feel like an executive operational coordination system, not a sci-fi cyber visualization.

Use:

- 70% executive operational clarity
- 30% restrained futuristic sophistication
- calm institutional command-center feel
- premium operational map aesthetic
- subtle topology and operational depth
- realistic/humanistic visual assets
- clean readable text
- meaningful motion only when useful

Avoid:

- cinematic spinning globe
- hacker-map spectacle
- cyberpunk theater
- heavy 3D animation
- AI stock-art energy
- generic SaaS blandness
- cartoon icons
- decorative bubbles

The original digital-world idea should be preserved only as restrained topology, not spectacle.

## Required visual behavior

- all assets must remain discoverable on desktop and mobile
- center canvas must support horizontal and vertical scroll when needed
- selected asset must come forward visually
- assets must not overlap
- connection lines must connect perimeter-to-perimeter, not run through assets
- relationship paths should visually route through CyberShield when appropriate
- hover state should clearly show interactivity
- click state should update a detail panel or route to a model/evidence/report
- MJC logo must appear in the header and center CyberShield shield unless explicitly changed

## Required TrustMap assets

Assets must be meaningful and premium.  Do not use placeholder icons.

### AI Systems and Agents

Use a brain or intelligence metaphor.  Do not use cute robot imagery.

### Modern Data Stack

Use a modern data stack visual distinct from Evidence Substrate.

### Evidence Substrate

Use proof/evidence/ledger/verified-records metaphor.  Do not reuse the same asset as Modern Data Stack.

### Policy and Compliance Manual

Use modern governance manual, policy binder, digital control book, or control library metaphor.  Do not use an old scroll.

### Brand and Identity Impersonation

Use a meaningful authenticity/impersonation visual that conveys fake claim, cloned identity, brand misuse, or trust attack.  Do not use a generic diamond.

### Third Parties and Vendors

Use a human/relationship/vendor ecosystem metaphor.  Avoid cartoon handshake imagery.

### Digital Identity

Use serious digital identity/authority metaphor.  A generic lock is insufficient unless upgraded into a higher-quality identity object.

## Required interaction behavior

Every TrustMap node must route to:

- node meaning
- owner
- related score model
- related evidence
- score impact
- recommended action
- report/export path

Every TrustMap edge must route to:

- relationship meaning
- risk if weak
- evidence needed
- model impact
- decision consequence
- report/export path

## Cross-domain impersonation rule

Brand and Identity Impersonation must be treated as cross-domain risk, not an isolated node.

It connects to:

- domains
- vendors
- sellers
- payment destinations
- product claims
- marketplace listings
- social/ad platforms
- customer trust
- legal/IP response
- evidence
- decision records
- executive authority

## Wow-surface requirement

TrustMap must create an immediate executive realization, not merely visual interest.

The user should think:

> This could absolutely happen here.

And then:

> We do not currently have this level of operational visibility.

## Performance requirement

TrustMap must remain lightweight enough for Firefox and mobile use.  Do not introduce heavy animation or rendering that slows the browser.

## QA checklist

Before accepting any TrustMap rebuild, verify:

- TrustMap loads quickly in Firefox
- all nodes are visible or discoverable through scroll
- no nodes overlap
- selected node comes forward
- edges connect cleanly
- hover previews work
- clicks update details or route deeper
- each node has model/evidence/report path
- no placeholder/cute/cartoon icons remain
- MJC logo appears in header and center shield
- visual style feels institutional, not sci-fi theater
- TrustMap creates a memorable operational realization
