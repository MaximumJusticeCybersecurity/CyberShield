Critical UX rule:
Every visible executive object that appears as a chip, metric, card, gate, node, report, status, or control must either explain itself on hover/focus or navigate somewhere useful. No dead bubbles.

Critical architecture rule:
CyberShield must remain consolidated and performant. Do not reintroduce the old phase-file module pile, broad MutationObserver loops, dynamic script injection, or timestamp-based runtime loading.

First-run UX rule:
First-run users see a streamlined Executive Trust Assessment before the dashboard appears. Dashboard navigation tabs remain hidden until onboarding is complete. After onboarding, the workspace appears.

Current onboarding model:
The onboarding flow remains five steps but each step collects multiple high-value fields:
1. Human in the loop: first name, organization, primary executive owner, first report audience
2. Role lens: primary executive decision lens
3. Operating environment: industry, business value at risk, primary executive goal
4. Evidence and ownership: evidence confidence, workflow maturity, AI use today, owner clarity
5. Dashboard shape: Executive, Security, Operations, or Board

Current runtime model:
Runtime is the center of gravity. The Runtime Control Center includes:
- Scenario queue
- AI action request
- Admissibility decision
- Trust score
- Evidence confidence
- Policy alignment
- Authority boundary
- Consequence model
- Why this decision panel
- Trust propagation
- Replayable evidence packet
- Runtime governance feed

Runtime scenarios currently included:
- AI vendor payment authorization
- AI-generated client communication
- Privileged access reset request
- Recovery validation automation
- Shadow AI sensitive data upload

Current TrustMap model:
The Operational TrustMap is not decorative. Runtime decisions propagate into the TrustMap. Current nodes include:
- Executive Authority
- AI Systems
- Runtime Rules
- Identity Boundary
- Vendor Access
- Critical Workflows
- Policy Boundary
- Evidence Substrate
- Executive Outputs

Current Reports model:
Reports use a three-pane workspace:
- Left: report catalog
- Center: generated report body
- Right: audience, recommended owner, and next conversation context

Report owners are assigned dynamically by report type and context, not hardcoded to one role.

Current Guidance model:
The prior sales language has been removed. The Guidance tab now uses executive continuity language:
- Current posture
- Recommended conversation
- What to prepare
- Personalized guidance note

CyberShield is not:
- a chatbot
- an LLM wrapper
- a generic AI dashboard
- a compliance spreadsheet
- an MSP portal
- a SIEM replacement
- a fully deployed enterprise telemetry platform in the current public prototype

CyberShield is:
- a runtime governance and operational admissibility prototype
- an Executive Advisor Layer for the human in the loop
- a Decision Intelligence Layer in architecture notes
- a pre-execution control-plane concept for AI-assisted and automated actions
- a system for showing decision outcome, rationale, owner, evidence, consequence, trust propagation, and escalation path
- a proof engine for Maximum Justice Cybersecurity advisory services

Public prototype boundary:
The current GitHub Pages build is a lightweight static prototype. It is not yet connected to live enterprise telemetry, real identity systems, SIEM data, EDR, cloud control planes, GRC tooling, or production agent enforcement. Do not represent it as a fully deployed live enforcement engine.
